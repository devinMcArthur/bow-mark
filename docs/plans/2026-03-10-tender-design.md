# Tender Feature Design

**Date:** 2026-03-10
**Branch:** `feature/tender`
**Status:** Approved

## Overview

A "Tender" represents a job that Bow-Mark has received and is in the process of bidding or estimating. Users can attach all documents related to the tender (specs, drawings, schedules of quantities, geotechnical reports, addendums, etc.), provide a written description, and chat with an AI assistant that has full access to those documents.

Primary use cases:
- **Bidding & estimating**: Understand the job scope, synthesize requirements across documents, assist in preparing proposals
- **On-site reference**: Foremen and project managers can ask questions about specs and requirements without manually searching through hundreds of pages

Access is restricted to **Project Manager** and **Admin** roles.

## Data Model

### `Tender` — new MongoDB collection

| Field | Type | Notes |
|---|---|---|
| `name` | `string` | Required |
| `jobcode` | `string` | Required, unique. Internal reference number e.g. `26-034`. Different numbering system from Jobsite jobcodes. |
| `description` | `string` | Optional writeup about the job |
| `status` | `enum` | `bidding` \| `won` \| `lost` |
| `jobsite` | `ref → Jobsite` | Optional. Set if bid is won and job enters active production. Tenders are never "converted" — they simply link to the Jobsite. |
| `files` | `EnrichedFile[]` | Subdocument array |
| `createdBy` | `ref → User` | |
| `createdAt` / `updatedAt` | `Date` | |

### `EnrichedFile` — subdocument on Tender

Named `EnrichedFile` (not `TenderFile`) so the schema can be reused on other models in the future.

| Field | Type | Notes |
|---|---|---|
| `file` | `ref → File` | Existing DO Spaces file model |
| `documentType` | `string` | Free text — e.g. "Spec Book", "Drawing", "Schedule of Quantities", "Geotechnical Report", "DSSP", "Addendum #2" |
| `summary` | `object` | AI-generated at upload time (see below) |
| `summaryStatus` | `enum` | `pending` \| `processing` \| `ready` \| `failed` |
| `pageCount` | `number` | |

**Summary object schema:**
```json
{
  "overview": "string",
  "documentType": "string",
  "keyTopics": ["string"]
}
```
Kept deliberately lightweight — its only purpose is to help Claude decide which documents to load. Detailed page-level information is retrieved from the full document at query time.

### `TenderConversation` — new MongoDB collection

Mirrors `ChatConversation` but scoped to a Tender rather than being user-global. Kept as a separate collection so the two chat features remain independent.

| Field | Type | Notes |
|---|---|---|
| `tender` | `ref → Tender` | |
| `user` | `ref → User` | |
| `title` | `string` | Auto-generated by Haiku on first turn |
| `messages` | `ChatMessage[]` | Same schema as `ChatConversation` |
| `totalInputTokens` / `totalOutputTokens` | `number` | |
| `createdAt` / `updatedAt` | `Date` | |

## Document Processing Pipeline

### On file upload

1. File is uploaded via existing mechanism → stored in DO Spaces → `File` document created
2. A `TenderFile` subdocument is added to the Tender with `summaryStatus: 'pending'`
3. A `tender.file.uploaded` event is published to RabbitMQ

### Background worker (RabbitMQ consumer)

Consumes `tender.file.uploaded` events:

1. Sets `summaryStatus: 'processing'`
2. Fetches the file from DO Spaces
3. Sends to **Claude Haiku** (cheap, fast, sufficient for structured extraction) with a prompt requesting:
   - Overview of the document
   - Document type
   - Key topics (materials, requirements, notable items)
4. For PDFs: passed as a native `document` content block — Claude sees the actual content including embedded images, tables, and drawings
5. For Excel: parsed to text/table representation using `xlsx`
6. Stores the structured summary, sets `summaryStatus: 'ready'`
7. On failure: sets `summaryStatus: 'failed'` — file is still usable in chat, just without a summary

## Chat Architecture

### Endpoint

`POST /tender-chat/message` — new Express router alongside existing `chat.ts`. Same streaming SSE pattern.

### What Claude receives each turn

**System prompt includes:**
- Tender name, jobcode, and description
- All `EnrichedFile` summaries — Claude's document index for deciding what to load
- Which files are still processing (if any), so Claude can caveat answers accordingly
- Instructions to cite sources inline using format `[Filename, p.X]`

**Tool — `read_document(file_id, start_page?, end_page?)`:**
- Claude calls this when a summary indicates a document is relevant and detail is needed
- PDFs: fetched from DO Spaces, passed as a native Anthropic `document` content block. Claude sees the actual content — drawings, embedded images, tables, annotations. No conversion, no OCR, no information loss.
- Excel: parsed to text/table representation
- `start_page` / `end_page` optional — Claude can request specific page ranges for large documents rather than loading the full file
- If a document would exceed context limits, the tool returns a warning and instructs Claude to narrow the page range

### Agentic loop

1. Claude reads all summaries from the system prompt
2. Claude determines which documents are relevant to the question
3. Claude calls `read_document` for 1–3 relevant files
4. Claude answers with inline citations: `[Spec Book Vol 1, p.3]`
5. Streaming SSE response sent to client

### Model routing

Same Haiku classifier pattern as existing chat:
- Simple lookups → **Sonnet**
- Complex synthesis, comparisons, multi-document reasoning → **Opus**

### Citations in the UI

`[Filename, p.X]` citations are parsed by the frontend and rendered as clickable links. Links open the PDF in a new tab using a DO Spaces pre-signed URL with a `#page=X` fragment — no custom PDF viewer needed.

### Conversation persistence

`TenderConversation` saved to MongoDB after each turn. Auto-generated title on first turn via Haiku (same pattern as existing chat).

## Frontend

### New pages

**`/tenders`** — list view
- Shows jobcode, name, status badge, file count, created date
- "New Tender" button
- Restricted to Project Manager and Admin roles

**`/tender/[id]`** — main Tender page with three sections:

**Overview**
- Tender name, jobcode, status badge, description
- Link to associated Jobsite if one is set
- Edit controls for basic fields

**Documents**
- File list: name, document type, page count, summary status badge
- Expandable row to preview the AI-generated summary
- Upload area — drag and drop, supports PDF, Excel, images
- Failed files show a retry button
- `Processing` badge on files still being summarised

**Chat**
- Sidebar: list of `TenderConversation` entries with "New conversation" button — same pattern as `/chat`
- Main area: streaming chat interface with citation link rendering
- Notice banner when files are still processing: "X documents still being processed — answers may be incomplete"

### Shared chat component

`ChatPage` is made configurable via props so it is shared between the global chat and tender chat — no duplicate code:

```tsx
<ChatPage
  messageEndpoint="/api/tender-chat/message"
  conversationsEndpoint="/api/tender-chat/conversations"
  extraPayload={{ tenderId: tender._id }}
  suggestions={TENDER_SUGGESTIONS}
/>
```

Improvements to the chat UI (streaming, citation rendering, markdown, cost display) automatically apply to both features.

### Navigation

"Tenders" entry added to the main nav sidebar, visible to Project Manager and Admin roles only.

## Error Handling & Edge Cases

| Scenario | Behaviour |
|---|---|
| Summary generation fails | `summaryStatus: 'failed'`, file still usable in chat without summary, retry button in UI |
| Chat started before all files processed | System prompt lists unprocessed files, Claude caveats answers accordingly |
| PDF exceeds context limits when loaded | Tool returns warning, Claude requests a specific page range |
| Tender has no files | Chat available, Claude's prompt notes no documents are attached |
| Corrupted / password-protected PDF | Clear error message on the file row in the UI |
| Concurrent uploads | Multiple files queued via RabbitMQ simultaneously, no blocking |

## Future Considerations

- **Cross-tender search**: Once multiple tenders exist, a RAG layer over all tenders could enable questions like "have we ever done a job with these spec requirements before?" — pgvector would be the natural fit at that point
- **EnrichedFile reuse**: The `EnrichedFile` subdocument schema can be adopted on `Jobsite` to bring AI document understanding to active jobs
- **Drawing quantity extraction**: Currently Claude can read annotated quantities from drawings but cannot calculate from geometry. Future work could integrate CAD parsing tools.
