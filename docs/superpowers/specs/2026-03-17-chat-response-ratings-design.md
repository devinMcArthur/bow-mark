# Chat Response Ratings Design

## Goal

Allow users to upvote or downvote AI assistant responses across all chat contexts (analytics, jobsite, tender). Downvotes capture a structured reason and optional comment. Data is stored for future use in an evaluation system.

## Scope

- All chat contexts: analytics, jobsite, tender
- No admin view in this iteration (planned separately alongside a Developer role)
- Users can change or clear their rating at any time

---

## Data Model

Add three optional fields to `ConversationMessageClass` in `server/src/models/Conversation/schema/index.ts`:

```typescript
rating?: "up" | "down"
ratingReasons?: string[]
ratingComment?: string
```

The message `_id` is already generated by Mongoose for subdocuments. It must be included in all API responses that return messages (conversation detail endpoint).

### Valid reason values (enum)

- `wrong_answer` ‚Äî factually incorrect based on the documents
- `hallucinated_citation` ‚Äî referenced a page/section that doesn't say what it claims
- `couldnt_find_it` ‚Äî answer exists in the documents but Claude said it wasn't there
- `wrong_document` ‚Äî answered from the wrong file or section
- `too_vague` ‚Äî hedged too much, didn't give a concrete answer
- `misunderstood_question` ‚Äî answered a different question than what was asked

## API

### PATCH /api/conversations/:id/messages/:msgId/rating

Updates the rating on a specific message within a conversation.

**Auth:** JWT middleware (same as all other conversation routes)

**Request body:**
```json
{ "rating": "up" | "down" | null, "reasons": ["hallucinated_citation"], "comment": "Optional text" }
```

- `rating: null` clears an existing rating
- `reasons` and `comment` are only stored for downvotes; they are cleared server-side when rating is `"up"` or `null`
- At least one reason is required when `rating` is `"down"`

**Response:** `{ "success": true }`

**Validation:**
- Conversation must exist and belong to the requesting user
- `msgId` must match a message `_id` within the conversation
- `rating` must be `"up"`, `"down"`, or `null`
- If `rating` is `"down"`, `reasons` must be a non-empty array of valid reason values

**Note:** Both tender and jobsite conversations use the same `Conversation` collection, so this single endpoint covers all chat contexts.

## UI Components

### `RatingButtons` component

Rendered below each completed assistant message bubble, in the same action row as the Sources button and regenerate icon.

**Visibility:**
- On desktop: hidden by default, revealed on hover of the message container
- On mobile: always visible (no hover available)
- Only shown when `isStreaming: false`
- Never shown on user messages

**States:**
- Default: both thumbs shown in muted gray
- Upvoted: thumbs-up icon filled green, thumbs-down muted
- Downvoted: thumbs-down icon filled red, thumbs-up muted

**Interactions:**
- Click thumbs-up: submit `rating: "up"`, update local state optimistically
- Click active thumbs-up again: submit `rating: null` to clear
- Click thumbs-down: open `DownvoteForm` inline
- Click active thumbs-down again: re-open `DownvoteForm` pre-filled with existing reasons

**Icons:** `FiThumbsUp` / `FiThumbsDown` from `react-icons/fi` (already used in project)

### `DownvoteForm` component

Expands inline below the action row when thumbs-down is clicked.

**Contents:**
- Header: "What was wrong with this response?"
- Six checkboxes, one per reason category (labels are human-readable, values are the enum keys)
- Optional comment textarea (placeholder: "Optional comment...")
- Cancel and Submit buttons

**Behavior:**
- Submit is disabled until at least one checkbox is selected
- On submit: call rating API, collapse form, show red thumbs-down as active
- Cancel: collapse form without saving; if a downvote was already saved, leave it as-is
- When re-opening an existing downvote, pre-fill checkboxes and comment with saved values

## Client-Side State

### `ChatMessage` type changes (`client/src/components/Chat/types.ts`)

Add three optional fields:
```typescript
messageId?: string      // MongoDB _id of the message subdocument
rating?: "up" | "down"
ratingReasons?: string[]
```

### Loading a conversation

`GET /api/conversations/:id` now includes `_id`, `rating`, `ratingReasons`, and `ratingComment` per message. The client maps these onto `ChatMessage` objects so rated state is restored on load.

### Rating a message

`ChatPage` passes a `rateMessage(messageId, rating, reasons?, comment?)` callback down to `MessageBubble` ‚Üí `RatingButtons`. On call:
1. Optimistically update local message state
2. Fire `PATCH /api/conversations/:id/messages/:msgId/rating`
3. On error: revert local state

### Streaming

`RatingButtons` is not rendered while `isStreaming: true`. It appears once the stream completes and `isStreaming` is set to `false`.
